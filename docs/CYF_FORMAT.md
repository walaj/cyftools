# CYF Format Specification

**Version:** 1.0

A reference implementation accompanies this document at [`docs/reference/cyf.py`](reference/cyf.py); the byte offsets in the worked example (§11) are generated by it.

---

## 1. Overview

CYF stores single-cell measurement tables from multiplex imaging (e.g. CyCIF, Orion, mIF). A CYF file has a header section followed by a record section. Each record is one cell. The header declares the column schema and the meaning of every flag bit, so a file is interpretable from itself alone.

The format defines two interchangeable encodings of the same logical content, after SAM/BAM:

- a TAB-delimited **text** encoding (`.cyf`), and
- a **binary** encoding wrapped in BGZF (`.byf`).

Conversion between the two is lossless in both directions (§9). All data columns are typed; types are declared once in the header (§3), and each record stores bare positional values in header order (§5). Unlike BAM optional fields, which carry a per-record type code, CYF binds types in the header — closer to a fixed table schema.

Throughout this document, the key words *must*, *should*, and *may* are to be read as requirement levels.

---

## 2. Conceptual model

```
  HEADER   ordered list of tag records (the schema)
             HD  format version, sort order
             SA  sample / specimen
             MA  marker        ─┐
             CA  metadata        ├─ define data columns, in header
             GA  graph         ─┘   order; this order is the schema
             FL  flag-bit semantics (cflag / pflag bits)
             PG  program / provenance chain

  RECORDS  cell, cell, cell, …   positional typed values per the schema
```

The ordered set of `MA` + `CA` + `GA` tags is the column schema. The *k*-th data value in every record corresponds to the *k*-th data tag in header order, encoded according to that tag's declared type.

---

## 3. Type system

Every data column declares a single-character **type code**. Codes follow BAM's optional-field types.

| Code | Meaning | Binary encoding | Text rendering |
|------|---------|-----------------|----------------|
| `i` | signed 32-bit int | 4 bytes, little-endian | decimal |
| `I` | unsigned 32-bit int | 4 bytes, LE | decimal |
| `l` | signed 64-bit int | 8 bytes, LE | decimal |
| `L` | unsigned 64-bit int | 8 bytes, LE | decimal |
| `f` | 32-bit float (IEEE-754) | 4 bytes, LE | shortest round-trip decimal |
| `d` | 64-bit float (IEEE-754) | 8 bytes, LE | shortest round-trip decimal |
| `Z` | UTF-8 string | `u32` length + that many bytes | raw text (tabs/newlines escaped) |
| `A` | categorical / enum | `u32` index into the column's `LV:` level list | the level string |
| `B` | typed array | `u8` subtype code + `u32` count + count×subtype values | `<sub>:v0,v1,…` (e.g. `L:2,3,4`) |

Notes:

- Every data column has a type. A reader treats a missing `TY:` as `f` (float) for
  backward compatibility, but a **writer should emit an explicit `TY:` on every data
  column** (`cyftools convert` stamps `TY:f`; `cyftools settype` sets any other type).
- `MA` (marker) columns default to `f` when `TY:` is omitted, and may declare any numeric type.
- `A` (categorical) is the representation for **label columns that must not be treated as
  numbers** — a cell-type name, or an integer cluster id (e.g. an immune-cluster `IC`).
  It stores a small integer index on disk; the labels are held in the column's `LV:` field
  (e.g. `LV:tumor,stroma,immune`, or `LV:0,1,2,17,28` for integer cluster ids). The index
  is a position into `LV:`, **not** the label value, so non-contiguous integer labels are
  fine. `cyftools settype <in> <out> --set <col>:A` builds `LV:` from the column's distinct
  values and rewrites each record's index.
- `B` (array) carries a variable-length per-cell vector. Its subtype must be one of the fixed-width numeric codes (`i I l L f d`).
- Missing values: for `f`/`d`, missing is IEEE NaN. For integer and `A` columns, a column may declare `NA:<value>` in its header tag to reserve a sentinel; absent that, all values are significant.

## 4. The header

### 4.1 Grammar

The header is a sequence of **tag records**, one per line in the text form. Each tag record is:

```
@<CLASS> <TAB> <KEY>:<VALUE> <TAB> <KEY>:<VALUE> …
```

- `<CLASS>` is a two-letter uppercase code: `HD`, `SA`, `MA`, `CA`, `GA`, `FL`, `PG`.
- Each field is a two-or-more-letter `KEY`, a colon, then the value. Keys are unique within a record. Field order within a record is preserved on write; `ID`, when present, comes first.
- `VALUE` may contain spaces but not raw tabs or newlines; those are escaped as `\t`, `\n`, `\\` in text and stored decoded in the binary header block.

### 4.2 Tag reference

#### `@HD` — file header (required, must be first)

Exactly one, the first line.

| Key | Req | Meaning |
|-----|-----|---------|
| `VN` | yes | format version string, e.g. `1.0` |
| `SO` | no | sort order: `unsorted` (default), `coordinate`, `id` |
| `MP` | yes | microns per pixel (positive float) — the coordinate scale |
| `UN` | yes | units of the `x`/`y` record fields: `micron` or `pixel` |
| `CW` | no | flag register width in bits; v1 fixes this at `64` |

`MP` and `UN` make every file self-describing about its coordinate scale. They
are stamped by `cyftools convert` (which **requires** `-c <mpp>` and
`-u <micron|pixel>`). Run `cyftools validate <file>` to check a file carries
them; add them to an older file with
`cyftools addtag in out -t HD -f MP:<mpp> -f UN:<micron|pixel>`.

#### `@SA` — sample / specimen

Zero or more.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | sample id; the integer carried in the high 32 bits of every record `id` (§5) |
| `SM` | no | human sample name |
| `SL` | no | slide id |
| `PA` | no | panel name (e.g. `Orion`, `prostate-12plex`) |
| `DS` | no | free-text description |

#### `@IM` — image / acquisition

Acquisition metadata for the image a sample was quantified from. Zero or more; one per sample.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | sample id this image belongs to (matches an `@SA ID` and the record `id` high bits) |
| `TF` | no | source image filename (e.g. the OME-TIFF) |
| `TH` | no | source image hash (md5 by convention) |
| `MP` | no | microns per pixel (spatial calibration) |
| `MS` | no | microscope / instrument |
| `DS` | no | free-text description |

#### `@RO` — region of interest

A spatial region (e.g. an annotation polygon) stored in the header, so it travels with the file without adding per-cell columns. Zero or more.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | unique region id |
| `NM` | no | human name (may contain spaces and commas) |
| `SA` | no | sample id the region applies to (default 0) |
| `TY` | no | shape type: `Polygon` (default), `Rectangle`, `Ellipse` |
| `PT` | for `Polygon` | vertices as space-separated `x,y` pairs: `x1,y1 x2,y2 …` (image coordinates) |

Because header fields are split on tab only (§4.1), the `PT` list and the `NM` name carry spaces and commas verbatim; no escaping is needed beyond the tab/newline rule. Non-polygon shapes may instead carry geometry keys such as `CX`/`CY` (center), `RX`/`RY` (radii), or `W`/`H` (size).

#### `@MA` — marker (defines a data column)

One per measured marker; order defines column position.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | marker name (e.g. `CD3`, `DAPI`) |
| `TY` | no | type code (default `f`) |
| `CH` | no | imaging channel |
| `EX`/`EM` | no | excitation / emission wavelength |
| `AB` | no | antibody / clone |
| `UN` | no | measurement units |
| `DS` | no | description |

#### `@CA` — cell annotation / metadata (defines a data column)

Per-cell values that are not raw marker intensities.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | column name (e.g. `nucleus_area`, `celltype`) |
| `TY` | yes | type code |
| `LV` | if `TY:A` | comma-separated category levels; record stores the index |
| `NA` | no | reserved missing-value sentinel (non-float types) |
| `PG` | no | id of the `@PG` program that created this column |
| `DS` | no | description |

#### `@GA` — graph column (defines a data column)

A per-cell spatial-graph payload, normally a `B` array.

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | column name (e.g. `knn_ids`) |
| `TY` | yes | type code, normally `B` |
| `KD` | no | graph kind: `knn`, `delaunay`, `radial` |
| `K` | no | neighbor count / parameter |
| `DS` | no | description |

#### `@FL` — flag-bit semantics

Declares the meaning of a single bit in the `cflag` or `pflag` register. A reader uses `@FL` records to label flag bits without external information.

| Key | Req | Meaning |
|-----|-----|---------|
| `RG` | yes | register: `cflag` or `pflag` |
| `BI` | yes | bit index, 0-based |
| `ID` | yes | symbolic name (e.g. `TUMOR`, `CD3pos`) |
| `DS` | no | description |

Example: `@FL␉RG:pflag␉BI:11␉ID:CD3pos` declares that bit 11 of `pflag` denotes CD3-positive.

#### `@PG` — program / provenance

| Key | Req | Meaning |
|-----|-----|---------|
| `ID` | yes | unique invocation id |
| `PN` | no | program name (`cyftools`) |
| `VN` | no | program version |
| `CL` | no | full command line |
| `PP` | no | `ID` of the previous `@PG` (chain link) |
| `DS` | no | description |

### 4.3 Ordering rules

- `@HD` is first.
- The relative order of `MA`/`CA`/`GA` records defines the data-column order and therefore the record value order. It must be stable across a round-trip.
- `@PG` records are emitted in chain order (oldest first).
- Other classes may appear in any position after `@HD`; writers should group by class for readability.

## 5. The record (logical)

A record (one cell) has five **mandatory fields** followed by one value per data column:

| Field | Type | Notes |
|-------|------|-------|
| `id` | `u64` | packed identifier: `(sample_id << 32) | cell_id` (see below) |
| `x` | `f32` | x coordinate |
| `y` | `f32` | y coordinate |
| `cflag` | `u64` | cell/structure flag register; bit meanings from `@FL RG:cflag` |
| `pflag` | `u64` | phenotype flag register; bit meanings from `@FL RG:pflag` |
| *data…* | per schema | one value per `MA`/`CA`/`GA` tag, in header order |

The five mandatory fields are always present, even when all data columns are absent.

**Identifier packing.** `id` is a `u64` split into two 32-bit halves:

```
 63                              32 31                               0
+----------------------------------+----------------------------------+
|            sample_id             |             cell_id              |
+----------------------------------+----------------------------------+
```

- `sample_id` (high 32 bits) names the sample/specimen and keys into the `@SA` header records (§4.2). A single-sample file uses `sample_id = 0`.
- `cell_id` (low 32 bits) is the per-sample cell identifier.

Because the sample occupies the high bits, the full 64-bit `id` is **globally unique across samples**: cells from different samples never collide even when their `cell_id`s coincide, so a file MAY hold multiple samples and all per-record operations (sort, distance, filtering) act on globally-unique ids. The encoding caps a file at 2³²−1 samples of up to 2³²−1 cells each. The EOF sentinel value (§7.3) remains an illegal `id`. Readers extract the halves as `sample_id = id >> 32` and `cell_id = id & 0xFFFFFFFF`.

**Coordinate precision.** `x`/`y` are `f32`. `f32` resolves to ≤ 1 µm out to ~16 million µm and to ~0.01 µm out to ~160 mm. A dataset needing higher precision may carry coordinates as `d`-typed `CA` columns.

**Flag width.** `cflag`/`pflag` are fixed at 64 bits.

**Phenotype-bit convention.** Bit *i* of `pflag` corresponds to the *i*-th `@MA` marker (0-based, in header order): the first marker is bit 0, the second bit 1, and so on. `@CA`/`@GA` columns do not consume bits. The marker order in the header is therefore the single source of truth for the phenotype-bit layout — no separate legend is required. Tools that set phenotype flags (`cyftools pheno`) and any reader that interprets them use this mapping. (`@FL RG:pflag` tags, if present, are advisory only and must not contradict it.)

## 6. Text format

A text CYF file is the header (the `@`-lines of §4) followed by one record per line. Each record line is **TAB-delimited**:

```
id <TAB> x <TAB> y <TAB> cflag <TAB> pflag <TAB> data0 <TAB> data1 <TAB> …
```

- `cflag`/`pflag` render as unsigned decimal integers. A writer may offer hex (`0x…`) output, but decimal is canonical.
- Float fields use the shortest decimal string that round-trips to the same IEEE value.
- `Z` strings are emitted with tabs, newlines, and backslashes escaped.
- `A` categoricals may render as the level string (canonical) or the integer index (accepted on read).
- `B` arrays render as `<subtype>:v0,v1,…` (e.g. `L:2,3,4`), or an empty field for a zero-length array.

---

## 7. Binary format

All multi-byte integers and floats are **little-endian**. The binary stream is a fixed preamble, then a stream of records, then an EOF marker. In a stored file this stream is wrapped in BGZF (§10).

### 7.1 Preamble

| Offset | Field | Type | Value / meaning |
|--------|-------|------|-----------------|
| 0 | `magic` | `char[4]` | `43 59 46 01` = `"CYF\x01"` |
| 4 | `version` | `u16` | format major version (1 for this spec) |
| 6 | `flags` | `u16` | layout flags (was reserved=0). Bit 0 = **reheader-friendly** (§10.1) |
| 8 | `l_header` | `u32` | byte length of the header text block |
| 12 | `header_text` | `char[l_header]` | the full text header (the `@`-lines), UTF-8, newline-separated, stored verbatim. May be padded past the last `@`-line with filler (any non-`@` lines), which readers ignore. |
| 12+`l_header` | `n_dcol` | `u32` | number of data columns |
| … | `dcol_types` | `u8[n_dcol]` | the type code of each data column, in header order |

The header is stored as the same UTF-8 text used by the text format, making text↔binary conversion lossless. The `dcol_types` array lets a reader decode records by offset arithmetic without parsing the header text, and lets a reader verify that its header parse agrees with the writer.

The `flags` field was a reserved zero in earlier writers; readers must ignore unknown bits, and a zero value is always valid (it just means the fast-reheader optimization below is unavailable). Padding the `header_text` is invisible to any reader: `l_header` covers the padding and the header grammar skips non-`@` lines.

### 7.2 Record stream

Each record is written back-to-back with no separator:

| Field | Type |
|-------|------|
| `id` | `u64` |
| `x` | `f32` |
| `y` | `f32` |
| `cflag` | `u64` |
| `pflag` | `u64` |
| each data value | encoded per its `dcol_types[k]` (§3) |

Fixed-width columns make a record fixed-size; `Z`/`A`/`B` columns make it variable-size, but always self-delimiting given the schema.

### 7.3 End-of-file marker and truncation detection

A well-formed stream ends with the 8-byte marker:

```
EOF_MARKER = 43 59 46 01 45 4F 46 00   ("CYF\x01EOF\x00")
```

A reader sits at a record boundary (having decoded a complete record using the known schema) and reads the next 8 bytes:

- if they equal `EOF_MARKER`, this is a clean end of file;
- otherwise, those 8 bytes are the next record's `id`; continue.

The `u64` value of the marker — **19790406162471235** — is a reserved, illegal cell id. Writers must never emit a cell with that id. A stream that ends without the marker is truncated or corrupt, and a reader must report it rather than treat it as a clean end.

## 8. Versioning and compatibility

- The binary `version` (`u16`) and the text `@HD VN:` carry the **format** version, independent of the program version (which lives in `@PG VN:`).
- A **major** version bump signals an incompatible byte layout; a reader must refuse a major version it does not implement.
- **Minor** additions (new optional header keys, new type codes used only when present) must be backward-compatible: an older reader skips unknown header keys and should error only when it encounters a data column whose type code it does not understand.

---

## 9. Lossless conversion

The binary preamble stores the header as the identical UTF-8 text used by the text format, and both encodings carry the same typed values. Conversion is therefore lossless in both directions:

```
text   ──parse header, parse rows──▶  binary
binary ──emit verbatim header, format rows──▶  text
```

A conformant tool must guarantee that `text → binary → text` and `binary → text → binary` are identity transforms, modulo the canonical float rendering of §6.

## 10. File forms, compression, and indexing

CYF mirrors SAM/BAM's two-tier model:

| Tier | SAM/BAM | CYF | Extension | Compressed |
|------|---------|-----|-----------|------------|
| Text | SAM | text (§6) | `.cyf` | no — TAB-delimited |
| Binary | BAM | binary (§7) wrapped in BGZF | `.byf` | yes |

The §7 byte stream is not a stored file form on its own; like BAM, it is the payload inside BGZF blocks. A stored file is either the **text** form (`.cyf`) or the **BGZF binary** form (`.byf`).

An implementation selects the form to write from the output extension: `.cyf` → text, `.byf` → binary; an output with no recognized extension defaults to binary. The extension `.ocyf` is reserved for an implementation's prior serialization and is read-only. Reading is content-detected and does not depend on the extension.

- **BGZF container.** The §7 byte stream is wrapped, unchanged, in BGZF — blocked gzip with a `BC` extra field per block and the standard 28-byte EOF block. A `.byf` file begins with the gzip magic `1f 8b`; decompressing it yields the §7 stream, beginning with the `CYF\x01` magic. Because it is gzip, `.byf` is readable by `zcat`/`bgzip`. Readers detect the encoding from the leading bytes (`@` → text; `1f 8b` → BGZF; `CYF\x01` → bare binary). BGZF also carries the virtual file offsets (block offset + in-block offset) used for random access.
- **Index (`.cyi`, deferred).** A coordinate index (binning the slide in x/y, after `.bai`/`.csi`) or a sample/id index, built on BGZF virtual offsets, would allow fetching the cells in a region or sample without a full scan. The `@HD SO:` sort-order field is the hook that makes an index meaningful.

### 10.1 Reheader-friendly layout (fast reheader)

When the preamble `flags` bit 0 is set, the writer has laid the file out so the header can be replaced without touching the cell records — the same trick `samtools reheader` uses for BAM:

- The `header_text` is **padded** to a capacity (the writer rounds up to a 64 KiB granularity), so a replacement header has room to grow without re-laying-out the file.
- A **BGZF block boundary is flushed immediately after the preamble** (`magic`…`dcol_types`), so the cell records begin on their own BGZF block(s).

A reheader then walks BGZF blocks until the cumulative uncompressed size equals the preamble size, writes new header block(s), and **copies the remaining compressed record blocks verbatim** (followed by a fresh EOF block) — no cell is decoded. This requires the column schema (`n_dcol`/`dcol_types`) to be unchanged; header-only commands (`addtag`, `addroi`, `clearroi`, `scaleroi`) qualify, and fall back to a full streaming rewrite for files without the flag. Because the padding and flag are transparent (§7.1), files written this way are read identically by older readers.

## 11. Worked example

Generated by `docs/reference/cyf.py`. The header (10 tag records) and two cells:

```
@HD	VN:1.0	SO:unsorted
@SA	ID:S1	SM:tonsil_01	PA:Orion
@MA	ID:DAPI	TY:f	CH:1
@MA	ID:CD3	TY:f	CH:5
@CA	ID:nucleus_area	TY:i
@CA	ID:celltype	TY:A	LV:tumor,stroma,immune
@GA	ID:knn_ids	TY:B	KD:knn	K:3
@FL	RG:cflag	BI:0	ID:TUMOR
@FL	RG:pflag	BI:1	ID:CD3pos
@PG	ID:pg1	PN:cyftools	VN:2.0	CL:convert in.csv
1	100.5	200.25	1	2	12.0	3.5	42	2	L:2,3,4
2	101.0	202.0	0	0	8.0	0.0	37	1	L:1,4,5
```

The binary encoding of the same content is 484 bytes. Annotated hexdump of the preamble and first record (offsets in decimal):

```
   0: 43 59 46 01                       magic 'CYF\x01'
   4: 01 00                             version u16 = 1
   6: 00 00                             reserved u16
   8: 2d 01 00 00                       l_header u32 = 301
  12: <301 bytes of UTF-8 header text>
 313: 05 00 00 00                       n_dcol u32 = 5
 317: 66 66 69 41 42                    dcol type codes = "ffiAB"

 --- first record ---
 322: 01 00 00 00 00 00 00 00           id u64 = 1
 330: 00 00 c9 42                       x f32 = 100.5
 334: 00 40 48 43                       y f32 = 200.25
 338: 01 00 00 00 00 00 00 00           cflag u64 = 1   (bit0 TUMOR set)
 346: 02 00 00 00 00 00 00 00           pflag u64 = 2   (bit1 CD3pos set)
 354: 00 00 40 41                       DAPI  f32 = 12.0
 358: 00 00 60 40                       CD3   f32 = 3.5
 362: 2a 00 00 00                       nucleus_area i32 = 42
 366: 02 00 00 00                       celltype A idx = 2  -> "immune"
 370: 4c                                knn_ids B subtype 'L' (u64)
 371: 03 00 00 00                       knn_ids B count = 3
 375: 02 00 00 00 00 00 00 00           knn_ids[0] = 2
      …                                 knn_ids[1], knn_ids[2], then record 2, then EOF_MARKER
```

The categorical `celltype` stores the index `2` on disk while the header's `LV:tumor,stroma,immune` supplies the label; the `knn_ids` graph column carries a self-describing typed array.
